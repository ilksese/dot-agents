# 海妖 Issue 工作流

## 任务目标

根据输入 issue 的类型自动选择工作流：

- `feature`：先完成需求分析、技术选型、方案制定和评审；用户通过后再实现，不通过则暂停。
- `bug`：先系统性定位根本原因，产出原因报告和修复方案；用户选择方案后再实现。

## 输入参数

- 原始输入：`$ARGUMENTS`
- issue 类型：从 `$ARGUMENTS` 中识别 `feature` 或 `bug`
- issue 内容：去除类型词后的剩余内容

## 类型识别规则

- 显式类型优先：`feature`、`bug`。
- feature 别名：需求、功能、新功能、优化、改造。
- bug 别名：缺陷、问题、报错、异常、修复、debug。
- 如果无法判断类型，先询问用户：这是 feature 还是 bug？不要继续后续流程。
- 如果同时出现 feature 和 bug 信号，优先按用户显式写出的 `feature` 或 `bug` 判断；仍有歧义时先询问用户。

## 通用规则

- 在用户明确通过（approve）最终方案前，禁止修改、创建、删除或格式化任何文件；只能阅读、分析、运行安全的只读命令和输出方案。
- 修改项目代码前，先阅读目标项目内适用的 `AGENTS.md` 或项目说明。
- 优先复用项目已有技术、组件、工具和模式；不要为了单个 issue 新增依赖或抽象。
- 不要读取任何 `.env*` 文件。
- Web 服务必须使用 `pm2` 管理。
- 需要验证时，只运行项目允许的最小相关验证命令；禁止执行项目规则禁止的构建或 `tsc` 命令。

## Feature 工作流

按顺序执行：

1. 需求分析
   - 提炼用户目标、使用场景、输入输出、边界条件和验收标准。
   - 信息不足但可安全默认时，给出默认假设继续；会 materially 改变结果时再提问。
2. 技术选型
   - 优先复用现有实现、组件、hook、工具函数、接口和样式体系。
   - 只有现有项目无法覆盖且收益明确时，才提出新增依赖或新抽象。
3. 方案制定
   - 输出最小可落地方案、涉及文件/模块、数据流、风险点和验证方式。
4. 方案评审（自动）
   - 自查方案是否过度设计、是否影响无关模块、是否有更小改动、是否满足验收标准。
   - 自动评审通过后，继续用户评审；不通过则修正方案后重新自查。
5. 方案评审（用户）
   - 向用户展示最终方案并等待确认。
   - 用户通过后进入落地实现。
   - 用户不通过或要求调整时，暂停实现流程，并按反馈更新方案。
6. 落地实现
   - 只实现已通过方案中的内容。
   - 完成后运行最小相关验证，并说明结果。

## Bug 工作流

按顺序执行：

1. 问题复现与边界确认
   - 明确复现路径、环境、端类型、账号/登录态要求、期望行为和实际行为。
   - 需要登录时，让用户在浏览器中手动登录；用户回复已登录前不要继续。
2. 系统性根因定位
   - 从报错、日志、网络请求、路由、状态、数据流、组件渲染、缓存、权限、边界输入和最近变更中逐项排查。
   - grep/CodeGraph 检查相关函数、组件、接口和调用方，不只看 issue 中点名的文件。
   - 区分触发症状、直接原因和根本原因。
3. 相似问题排查
   - 搜索同类逻辑、同名字段、相同 API、相同状态处理、相同组件模式或复制实现。
   - 标记哪些属于 issue 影响模块，哪些属于项目级同类风险。
4. 原因报告
   - 输出复现结论、影响范围、根本原因、证据、相关文件/模块和为什么不是其他原因。
5. 修复方案
   - 最小修复方案：只修改 issue 中影响的模块，改动最小，风险最低。
   - 项目级修复方案：在最小修复基础上，一并修复其他业务或文件中的同类问题。
   - 每个方案都说明涉及文件、改动点、风险、验证方式和推荐方案。
6. 用户选择
   - 在用户选择最小修复方案或项目级修复方案前，不要修改代码。
   - 用户选择后，只实现对应方案并运行最小相关验证。
